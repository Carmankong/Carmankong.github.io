    • Relevant Implications:

    Future proofing ---- Program is designed, so now features can be added in easily and old ones removed
    Addressed ----- Program element, (colours, buttons, shapes etc.) have all been programmed as an object, means each is a self-contained, modular piece of code, that takes a defined set of parameters to work. (Many take basic mouse data and colour information).
    All objects are built to a similar structure and compulsorily contain an update function while means they can appear on the interface using the update call.
    Management of most objects usually happens in a central place (control_draw.js).
    The idea is an experienced programmer could easily remove elements or add new features. Comments are also precise and informative.

    Functionality:

    Does the program work well?
    - Free of bugs, loads quickly, works on different browsers.
    How have we addressed this?
    - Program is in pure javascript. Has no graphics, therefore loads very quickly. I have tested this by using a live web server (Github) to store my version and have observed loading performance.
    To ensure the program works well, I have systematically tested it (console log, testing videos etc.)
    -Tested user interactions
    - Position of clicking and direction of dragging.
    - This has included careful checks around boundaries.
    - The use of the Math.abs function was discovered after testing failures drawing circular shapes. (because these objects were being given negative radius values).
    Any new components were tested prior to being added to the main program. At various stages in sprints, console logging was used to analyse boundary test functions, and locate problems when bugs appeared.

    Examples of trialling and making decisions:

    Can you find examples of exploding different options and making a decision about which is the best solution?
    -Trialling different options for allowing the user to draw circles on the screen.
    - Giving rectangular guides.
    Effectively using information from testing and trialling to improve the functionality of the outcome.
    Interface design trialling colour themes.
    Finding situations where objects drew outside of the drawing area (in spite of having in-boundary parameters) led to the use of the boundsCheck function.
    Good to talk about “project backlog” and the concept of prioritisation.
